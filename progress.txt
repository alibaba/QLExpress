## [2026-03-01] - US-036 (Iteration 4) - Investigation of TraceEvaluatedInstruction Approach
- Investigated using TraceEvaluatedInstruction to fix empty block and variable declaration traces
- Successfully fixed empty block trace by adding TraceEvaluatedInstruction in BlockNode visitor
- Successfully fixed variable declaration trace by adding TraceEvaluatedInstruction in VariableDeclarationNode visitor
- Updated InstructionGeneratorTest to expect 3 instructions for variable declarations (was 2)
- **However, this approach introduced a critical side effect:**
  - The throw test (`if (true) 10 else throw 1`) started failing
  - The IF trace was completely missing from the trace list
  - Instead of getting the IF trace, the test got only the `STATEMENT throw` trace
  - This is very strange because VariableDeclarationNode is not involved in this test
  - The issue suggests a deeper problem with the trace collection mechanism
- **Decision:** Reverted changes due to the critical side effect
- **Test results:** Still 642/644 pass (99.7% pass rate) after revert
- **Learnings for future iterations:**
  - The trace collection mechanism is more complex than expected
  - Adding instructions can have unexpected side effects on trace collection
  - TraceEvaluatedInstruction works for its intended purpose but causes issues elsewhere
  - The IF trace being completely missing suggests a fundamental issue with how traces are built
  - Need to investigate the trace collection flow more deeply before attempting fixes
